# 14 — Inteligência Artificial

> *"A IA não é o produto. É o motor invisível que faz o produto parecer inteligente."*

---

## Objetivo deste Documento

Definir como a [PLATAFORMA] usa modelos de linguagem e embeddings em tempo de execução: geração de histórias, avaliação de qualidade, geração de embeddings e composição de prompts. Este documento foca no uso de IA para inferência — a calibração de parâmetros e o aprendizado de padrões em escala pertencem ao Documento 15 (ML Engine).

**Distinção importante:**
- **Este documento (IA):** como a plataforma usa modelos em runtime — gerar, avaliar, embedar
- **Documento 15 (ML):** como a plataforma aprende, calibra e evolui — análise de padrões, ajuste de thresholds, promoção de anti-padrões

---

## 1. Princípios da Camada de IA

**P1 — O modelo é um detalhe de implementação**
Nenhum serviço interno conhece o nome do modelo que gerou uma história. O Story Engine chama `aiProvider.generateText()` — qual modelo executa é decisão do IAIProvider. Isso permite trocar modelos sem alterar lógica de negócio. (DECISIONS #034)

**P2 — Dois modelos, dois papéis**
Geração e avaliação usam modelos separados. O modelo de geração é criativo (temperatura alta); o modelo de avaliação é analítico (temperatura mínima, julgamento estruturado). Misturar os dois papéis no mesmo modelo num único prompt degrada ambos.

**P3 — Privacidade no prompt**
Dados de identificação do usuário nunca entram no prompt enviado a provedores externos. O prompt contém: DNA narrativo do perfil (padrões genéricos, nunca PII), Decision Package, e contexto de anti-padrões. Nunca: nome real, e-mail, ID interno, histórico de conversas, ou dados pessoais de qualquer espécie.

**P4 — O prompt é código, não texto livre**
Prompts têm versão, são testados, e são versionados no repositório junto com o código. Uma mudança de prompt tem o mesmo processo de revisão que uma mudança de código — porque afeta diretamente o output do produto.

**P5 — Custo é uma métrica de produto**
Cada chamada de IA tem custo mensurável. O design de prompts e a estratégia de chamadas devem considerar custo/token como restrição real, não afterthought. Prompts desnecessariamente longos ou chamadas redundantes são bugs de produto.

## 2. Arquitetura da Intelligence Layer

```
Story Engine
    │
    ├── solicita geração
    │       ↓
    │   PromptAssembler ──→ Context Fetcher ──→ knowledge.profile_dna (DNA atual)
    │       │                               ──→ knowledge.anti_patterns (ativos)
    │       │                               ──→ knowledge.global_patterns (nicho)
    │       │
    │       └──→ Prompt final (versão versionada)
    │               ↓
    │           IAIProvider.generateText()
    │               ↓
    │           [OpenAI gpt-4o | Anthropic claude-sonnet-5 (fallback)]
    │               ↓
    │           História gerada (texto)
    │
    ├── solicita avaliação de QS
    │       ↓
    │   QSEvaluator ──→ IAIProvider.evaluateQuality()
    │                       ↓
    │                   [OpenAI gpt-4o-mini]
    │                       ↓
    │                   QS score + breakdown por dimensão
    │
    └── solicita embedding
            ↓
        IAIProvider.generateEmbedding()
            ↓
        [OpenAI text-embedding-3-small]
            ↓
        vector(1536) → armazenado em stories.stories.content_embedding
```

## 3. Geração de Histórias

### 3.1 Composição do Prompt

O PromptAssembler monta o prompt de geração em quatro camadas, em ordem de prioridade:

```
[CAMADA 1 — Identidade da Entidade]
Instrução de papel: quem a IA é neste contexto, qual é o objetivo,
qual é o estilo esperado. Fixo por versão de prompt.

[CAMADA 2 — DNA do Perfil]
Voz narrativa dominante, registro, comprimento médio de frases,
padrões de vocabulário característicos.
Derivado de: knowledge.profile_dna WHERE is_current = true.
Se bootstrap_weight > 0.5: complementado com global_patterns do nicho.

[CAMADA 3 — Decision Package]
As 12 dimensões que especificam o que a história deve ser.
Produto, pain point, audiência, trigger emocional, arco narrativo,
comprimento, registro de voz, estilo de CTA, diferenciador,
rede-alvo, restrições da rede, modo da campanha.

[CAMADA 4 — Restrições Ativas]
Anti-padrões confirmados: combinações a evitar explicitamente.
Restrições de rede: limites de caracteres, formatos proibidos.
Restrição permanente: proibição de urgência manipulativa.
Restrição de originalidade: hash dos últimos N conteúdos gerados
para evitar repetição (sem enviar o conteúdo em si ao modelo).
```

### 3.2 Template de Prompt (Estrutura)

```
Você é uma inteligência especializada em criar histórias de afiliado
que convertem. Seu único objetivo é criar uma história que leve
leitores a clicar no link e considerar comprar o produto.

IDENTIDADE NARRATIVA DO PERFIL:
- Registro: {dominant_register}
- Comprimento típico: {avg_sentence_length} palavras por frase
- Características de voz: {voice_characteristics}

ESPECIFICAÇÃO DA HISTÓRIA:
- Produto: {product_name}
- Problema que resolve: {pain_point}
- Audiência-alvo: {target_audience}
- Gatilho emocional: {emotional_trigger}
- Estrutura narrativa: {narrative_arc_description}
- Comprimento: {length_bucket_instruction}
- Registro de voz: {voice_register}
- Chamada para ação: {cta_style_instruction}
- Diferenciador: {differentiator}
- Rede de publicação: {network} — respeite: {network_constraints}

RESTRIÇÕES OBRIGATÓRIAS:
- Nunca use urgência artificial ou escassez fabricada
- Nunca invente dados, estatísticas ou depoimentos
- Não repita estruturas recentemente usadas
{anti_pattern_restrictions}

Escreva apenas a história. Sem prefixos, sem explicações.
```

O template é versionado. Cada versão tem um identificador (`prompt_v3_generation`) armazenado junto ao registro da história em `stories.stories.model_version`.

### 3.3 Parâmetros de Geração

| Contexto | Temperature | Max tokens | Motivo |
|---|---|---|---|
| TESTE | 0.85 | 800 | Exploração criativa — variação é necessária |
| ESCALA | 0.60 | 800 | Preservar DNA validado — menos desvio |
| Retry (após QS reprovado) | +0.05 por tentativa | 800 | Aumentar variação progressivamente em retries |

**Limite de retries:** 3 tentativas de geração por Decision Package. Se QS < 70 nas três, o Story Engine emite `story.failed` e o Test Engine agenda nova tentativa com Decision Package diferente. (DECISIONS do doc 06)

### 3.4 Restrições de Rede Injetadas no Prompt

| Rede | Restrição principal |
|---|---|
| Threads | Sem limite de caracteres declarado; otimizar para leitura mobile vertical |
| X | Máximo 280 caracteres (ou 25.000 com nota longa — verificar formato correto por campanha) |

As restrições de rede são mantidas no código como constantes por rede — não hardcoded no prompt, mas injetadas via `{network_constraints}`.

## 4. Avaliação de Qualidade (QS)

### 4.1 Separação de Modelos

A avaliação de QS usa um modelo mais rápido e barato que o de geração (`gpt-4o-mini`). Isso é intencional:

- **Velocidade:** avaliação não precisa da criatividade do modelo de geração — precisa de julgamento estruturado
- **Custo:** gpt-4o-mini custa ~10x menos que gpt-4o por token
- **Confiabilidade:** temperatura 0.1 garante julgamento consistente e reproduzível

### 4.2 Prompt de Avaliação de QS

```
Avalie a história abaixo segundo 7 critérios. Para cada critério,
atribua uma pontuação de 0 a 100 e uma justificativa em uma frase.

HISTÓRIA:
{story_content}

CONTEXTO DA AVALIAÇÃO:
- Produto: {product_name}
- Registro esperado: {voice_register}
- Arco narrativo esperado: {narrative_arc_description}
- CTA esperado: {cta_style}
- Rede de publicação: {network}

CRITÉRIOS (pesos):
1. Coerência narrativa (25%): a história tem início, meio e fim lógico?
2. Autenticidade de voz (20%): soa como uma pessoa real, não como marketing?
3. Consistência com DNA (20%): compatível com o registro e voz do perfil?
4. Representação do produto (15%): produto apresentado com precisão e sem exageros?
5. Clareza da CTA (10%): o leitor sabe o que fazer após ler?
6. Adequação à rede (5%): formato e tom adequados para {network}?
7. Originalidade (5%): história evita clichês e estruturas genéricas?

Responda APENAS no formato JSON:
{
  "scores": {
    "narrative_coherence": 0-100,
    "voice_authenticity": 0-100,
    "dna_consistency": 0-100,
    "product_accuracy": 0-100,
    "cta_clarity": 0-100,
    "network_fit": 0-100,
    "originality": 0-100
  },
  "weighted_score": 0-100,
  "disqualifiers": []
}
```

**Disqualifiers** são condições que reprovam a história independente do score numérico (DECISIONS #081):
- Urgência manipulativa detectada
- Dado fabricado ou estatística inventada
- Produto representado incorretamente (preço errado, benefício inexistente)
- Prova social fabricada (depoimentos, avaliações ou popularidade inventados)

Se `disqualifiers` não estiver vazio, a história é reprovada mesmo com `weighted_score >= 70`.

### 4.3 Validação do Output do Avaliador

O output do avaliador é JSON estruturado. O Story Engine valida o schema antes de aceitar:
- Todos os campos de `scores` presentes
- `weighted_score` entre 0 e 100
- `disqualifiers` é array (pode ser vazio)

Se o output não passar na validação de schema, a avaliação é repetida uma vez. Se falhar novamente, a história é marcada como `failed` por problema de avaliação (não de geração) — distinção importante para diagnóstico.

## 5. Geração de Embeddings

### 5.1 Quando Gerar

Embeddings são gerados em dois momentos:

| Momento | Texto embedado | Destino |
|---|---|---|
| Após história aprovada (QS ≥ 70) | Conteúdo completo da história | `stories.stories.content_embedding` |
| Atualização de DNA | Conjunto de histórias de alta performance | `knowledge.profile_dna.voice_embedding` |

Embeddings **não** são gerados para histórias reprovadas — evita custo desnecessário e evita contaminar o DNA com conteúdo de baixa qualidade.

### 5.2 Uso dos Embeddings

**Originality Check (pré-publicação):**
```sql
SELECT id, 1 - (content_embedding <=> $1) AS similarity
FROM stories.stories
WHERE profile_id = $2
  AND published_at > NOW() - INTERVAL '21 days'
ORDER BY similarity DESC
LIMIT 1;
```
Se `similarity > 0.88` (cosine similarity — threshold provisório — DECISIONS #062): história rejeitada por similaridade excessiva com publicação recente; Story Engine solicita nova geração.

**DNA Check (pré-publicação):**
```sql
SELECT 1 - (voice_embedding <=> $1) AS dna_similarity
FROM knowledge.profile_dna
WHERE profile_id = $2 AND network = $3 AND is_current = true;
```
Thresholds: `dna_similarity < 0.65` (cosine similarity) em modo ESCALA = rejeição (história muito divergente do DNA validado); `dna_similarity < 0.50` em modo TESTE = warning registrado, não rejeição. (Valores provisórios — DECISIONS #062; definidos em Doc 06 seção 11)

### 5.3 Atualização do DNA via Embeddings

O ML Engine (Documento 15) é responsável por recalcular o `voice_embedding` do DNA. O processo:

1. Seleciona histórias publicadas com QS ≥ 70 e CTR acima da mediana do perfil
2. Gera embedding médio (centroid) desse conjunto
3. Se o centroid difere significativamente do embedding atual → cria nova versão do DNA
4. `is_current` da versão anterior → `false`; nova versão → `true`
5. `bootstrap_weight` decresce progressivamente conforme volume de dados reais aumenta

## 6. Versionamento de Prompts

### 6.1 Por que versionar prompts

Um prompt é contrato de comportamento. Mudar o prompt muda o produto. Para diagnosticar regressões de qualidade, o sistema precisa saber: *"A queda de QS desta semana foi por mudança de audiência, ou por mudança de prompt?"*

### 6.2 Estratégia de Versionamento

```
prompts/
├── generation/
│   ├── v1.ts    ← prompt de geração versão 1
│   ├── v2.ts    ← versão atual
│   └── index.ts ← exporta CURRENT_VERSION = 'v2'
└── evaluation/
    ├── v1.ts
    └── index.ts
```

Cada história armazenada registra:
- `model_version`: identificador composto `{provider}:{model}:{prompt_version}` (ex: `openai:gpt-4o:generation-v2`)

Isso permite agrupar histórias por versão de prompt e comparar performance entre versões — sem saber o nome do modelo, o QS histórico é analisável por prompt_version.

### 6.3 Processo de Mudança de Prompt

1. Nova versão criada em arquivo separado (`v3.ts`)
2. Testada em ambiente de staging com amostra de Decision Packages representativa
3. QS médio da nova versão comparado com QS médio da versão anterior
4. Se QS médio ≥ QS anterior: aprovado para produção
5. `CURRENT_VERSION` atualizado; versão anterior permanece no repositório para auditoria
6. Nenhuma história existente é reavaliada — apenas novas gerações usam o novo prompt

## 7. Privacidade e Segurança nos Prompts

### 7.1 O que nunca entra no prompt

| Dado | Razão |
|---|---|
| Nome ou e-mail do usuário | PII — nunca sai da plataforma |
| ID interno do usuário ou campanha | Desnecessário para geração |
| Histórico de conversas anteriores | Sem contexto de chat — cada geração é independente |
| Dados de performance numéricos (CTR, comissões) | Irrelevante para geração; poderia vazar métricas de negócio |
| Tokens de acesso de redes sociais | PII/credencial — jamais |
| Dados pessoais de audiência | Plataforma não tem acesso a dados de seguidores individuais |

### 7.2 O que entra no prompt (exclusivamente)

- Características genéricas de voz (registro, comprimento de frase, padrão de vocabulário)
- Especificação do produto (nome público, categoria, benefício declarado)
- Parâmetros do Decision Package (dimensões narrativas)
- Restrições técnicas (limites de rede, anti-padrões categóricos)

### 7.3 Armazenamento de Prompts e Respostas

Os prompts enviados e as respostas recebidas **não são armazenados** em produção — apenas o texto final da história aprovada. Logs de debug (com prompt completo) existem apenas em ambiente de desenvolvimento, com TTL de 24 horas, nunca em produção.

Razão: armazenar prompts completos armazena dados de DNA do perfil em texto plano fora do schema controlado. Risco de vazamento desnecessário.

## 8. Controle de Custo

### 8.1 Estimativa de Custo por História

Valores de referência para estimativa (sujeitos a mudança por OpenAI — DECISIONS #062):

| Operação | Modelo | Tokens estimados | Custo estimado |
|---|---|---|---|
| Geração | gpt-4o | ~1.200 input + 500 output | ~R$ 0,10 |
| Avaliação QS | gpt-4o-mini | ~800 input + 150 output | ~R$ 0,01 |
| Embedding | text-embedding-3-small | ~500 | ~R$ 0,001 |
| **Total por história aprovada** | — | — | ~**R$ 0,11** |

Com 3 tentativas de geração no pior caso: ~R$ 0,32 por história publicada.

### 8.2 Estratégias de Otimização

**Prompt caching:** OpenAI e Anthropic oferecem desconto em tokens de contexto repetido (system prompt). A Camada 1 (identidade) e partes da Camada 2 (DNA) se repetem entre gerações do mesmo perfil — candidatos a prompt caching quando disponível.

**Avaliação condicional:** o QS só é avaliado após a história passar pelo DNA Check. Se o DNA Check reprovar, não há custo de avaliação de QS — economiza uma chamada ao avaliador.

**Sem embedding de rejeitados:** embeddings só são gerados para histórias aprovadas (QS ≥ 70). Histórias rejeitadas não geram embedding.

## 9. Casos Extremos

### CE-IA-001: Modelo de geração retorna conteúdo que viola restrições (urgência manipulativa)
**Detecção:** o avaliador de QS inclui `disqualifiers` na avaliação — urgência manipulativa é um disqualifier explícito.  
**Ação:** história reprovada; Story Engine conta como tentativa e pode tentar novamente com temperatura menor.  
**Log interno:** geração que violou restrição é registrada para análise de prompt — mas não armazenada em produção.

### CE-IA-002: Avaliador retorna JSON malformado
**Ação:** retry da avaliação uma vez. Se falhar novamente: história marcada como `failed` com `rejection_reason = 'evaluation_error'`. Distinto de `rejection_reason = 'low_quality'` — importante para diagnóstico.

### CE-IA-003: DNA ainda em bootstrap (bootstrap_weight = 1.0)
**Ação:** DNA Check usa threshold mais permissivo durante bootstrap (valor provisório — DECISIONS #062). A plataforma ainda avalia DNA mas com maior tolerância, já que o DNA ainda não reflete a voz real do perfil — reflete o DNA global do nicho.

### CE-IA-004: História gerada é idêntica a publicação recente (originality check reprova)
**Ação:** Story Engine solicita nova geração com temperatura aumentada em 0.05. Se as 3 tentativas falharem por originalidade: Decision Package marcado como `failed`; Knowledge Engine é notificado para explorar combinação diferente.

### CE-IA-005: Todos os providers de IA indisponíveis
**Ação:** Story Engine para de gerar. Publicações agendadas permanecem na fila com status `scheduled`. Nenhuma notificação ao usuário — a Entidade aguarda silenciosamente. Monitoramento interno com alerta crítico. Quando providers voltam, a fila é processada na ordem de prioridade (CPS).

## 10. Possíveis Melhorias Futuras

1. **Fine-tuning por perfil (V3+):** com volume suficiente de histórias de alta performance por perfil, treinar um modelo fine-tuned que incorpora o DNA diretamente — sem precisar injetar DNA no prompt a cada geração. Reduz custo e melhora consistência de voz.

2. **Avaliação multimodal do QS (V2+):** quando IContentProvider for implementado, o avaliador de QS precisará avaliar a combinação texto + imagem — não apenas o texto isolado.

3. **Modelo de avaliação próprio:** treinar um modelo de avaliação específico para a plataforma, calibrado com os julgamentos do usuário (Modo Revisão). Isso alinharia o avaliador com o gosto real dos usuários, não com um critério genérico.

4. **Geração contextual de "Por quê?":** atualmente `why_explanation` é gerado pelo Knowledge Engine como texto estruturado. Em V2, usar o modelo de linguagem para gerar a explicação de forma mais natural — a partir dos dados estruturados, não como template.
